Módulo 05 - Documentar requisitos usando Linguagem Natural

Parte 01 - Efeitos da Linguagem 

Os requisitos são frequentementes documentados em linguagem natural(prosa livre).
A linguagem natural possui a vantagem de não exigir preparação para leitura pelas partes interessadas.

A linguagem natural pode ser utilizada para descrever quaisquer circunstâncias.
Não existe limites.

Desvantagem:

A lingaguem natural é inerentemente ambígua.
Exemplo:
	A mãe de Pedro entrou com seu carro na garagem.
	A mãe de Pedro entrou na garagem com o carro dela.

Partes interessadas possuem diferentes níveis de conhecimento, experiências e origens.

Durante o processo de percepção e representação das informações ocorrem os EFEITOS TRANSFORMACIONAIS que prejudicam o entendimento dos requisitos.

	EFEITOS TRANSFORMACIONAIS

1 - Nominalização
	
	* Na nominalização processos são reduzidos. 	
	* Na nominalização um processo é convertido em um evento e assim toda informação que descreve o processo é perdida.

Exemplo:
	No caso de uma falha do sistema, a reinicialização do sistema deve ser realizada.	

Os termos FALHA DO SISTEMA e REINICIALIZAÇÃO são processos que precisam ser analisados com mais critério.

	* Não há problemas em utilizar termos normalizados para descrever processos complexo. No entanto, o processo deve ser explicitamente definido pelo termo usado. A definição do processo é adicionada no glossário.
	* O termo normalizado não deve permitir mais de uma interpretação dos processos.
	* Não é necessário evitar nominalizações, porém, deve ser utilizado se o processo é completamente difinido.
	* Nominalizações devem ser examinadas se foram detalhadas suficientemente e se está claro para as partes interessadas.




2 - Substantivo sem indicador de referência
	* Sâo frequentementes especificados de forma incompleta.
	* Isso são chamados pelos linguistas de ausência de índice de refêrencia.
	* Exemplos de substantivos sem índice de refêrencia:
		* O usuário
		* O sistema
		* O controlador
		* A mensagem
		* Os dados
		* A função
	Exemplo:
Sem indicador de refêrencia:
Os dados devem ser exibidos para o usuário no terminal.

Perguntas:
	* Que DADOS são esses?
	* Quem é o USUÁRIO?
	* Que TERMINAL  é esse?

	Exemplo:
Com indicador de refêrencia:
O sistema de faturamento deve mostrar os dados de bilhetagem para o usuário registrado no terminal que ele estiver logado.

3 - Quantificadores universais
	* Quantificadores especificam quantidade e frequências.
	* Quantificadores universais agrupam um conjuto de objetos e fazem uma declaração sobre o comportamento desse conjunto. 
	* Quantificadores universais são utilizados com as seguintes perguntas:
		- Nunca
		- Sempre
		- Não
		- Nenhum
		- Cada
		- Todos
		- Alguns
		- Nada
	* Ao utilizar um quantificador universal existe o risco de que um comportamento ou propiedade não se aplicar a todos os objetos dentro do conjunto especificado.
	* Partes interessadas tendem a agrupar objetos, mesmo havendo exceções, onde o comportamento não se aplica a todos os objetos do grupo.
 
Exemplo:
 o sistema deve mostrar todos os dados em cada tela
	
Perguntas surgem:
	* Será TODOS os dados mesmo?
	* Devem ser mostrados em TODAS as telas?


4 - Condições Especificadas de Forma Incompleta
	* Uma condição formulada de forma incompleta é um indicador em potencial de perda de informação.
	* Um requisito que contém condições especificam geralmente apenas o comportamento que tem que ocorrer quando uma condição é alcançada.
	* É vital também definir o que deve ser feito quando a condição não é alcançada. Esta geralmente é omitida.
	* No caos de estruturas condicionais complexas, tabelas de decisão podem auxiliar o engenheiro de requisitos a encontrar variáveis não especificadas.
	* Palavras gatilhos
		- Se... Então
		- No caso de ...
		- Se ...
		- Em função de ...

Exemplo: 
	Os pedidos de venda acima de R$1.000 devem ser acatados pelo sistema de vendas.
	
As seguintes perguntas surgem:
	- O que fazer com os pedidos de R$1.000	?
	- O que faze com os pedidos inferiores a R$1.000?

 Usando de forma completa as condições:

	Os pedidos de venda acima de R$1.000 devem ser acatados pelo sistema de vendas  
		e
	Os pedidos com valor igual ou inferior a R$1.000 não devem ser acatados pelos sistema de vendas

5 - Verbos de processo especificados de forma incompleta
	* Alguns verbos de ação precisam de mais de um substantivo para serem completamente definidos.
	* O verbo transmitir, por exemplo, exige pelo menos três suplementos para ser considerado completo.
		- O que está sendo transmitido
		- De onde está sendo transmitido
		- Para onde está sendo transmito.


	* Em alguns casos os adjetivos ou advérbios podem ser utilizados para completar o sentido do verbo.
	Exemplo:
	Os dados foram transmitidos para um sistema de pedidos.

Perguntas?
	Que dados foram transmitidos?
	De onde os dados foram transmitidos?

 Os dados de pedido de vendas foram transmitidos do dispositivo móvel do vendedor para o sistema de pedidos.

	* Evitar o uso da voz passiva.
		Exemplo: Os dados de login foram fornecidos no login.
	* Ao utilizar voz ativa evita-se verbos formulados de forma incompleta.
		Exemplo: O usuário forneceu os seusdados no login.

Parte 02 - Construção de Requisitos usando Templates de Sentenças

Cruando Requisitos utilizando templates
	
	- Um template de requisito é arcabouço para a estrutura sintética de um requisito.
	- Templates de requisitos fornecem abordagem simples e fácil para reduzir os problemas de linguagem.
	- Template de requisitos auxiliam o engenheiro de requisitos em alcançar alta qualidade na definição de requisitos com baixo custo e bom aproveitamento do tempo. 
	- Para alcanççar clareza na construção de uma boa documentação deve-se utiliza templates de requisitos e o glossário.
 

Como utilizar corretamente templates de requisitos?

Passo 1. Determinar a obrigatoriedade legal:
	- Determina-se a obrigação legal de um requisito utiliza-se os seguintes verbos: deve, deveria, irá e pode.
	- Estes verbos identificam se os requisitos são obrigatórios, requisitos urgentes, requistos futuros e requisitos desejáveis.
 	- Em vez de se utilizar verbos pode-se utilizar um atributo de requisito para determinar a obrigação legal. 
	- Exemplo: Pedidos de venda com valor inferior a R$1.000,00 devem ser desconsiderados.
	

Passo 2. Determinar o núcleo do requisito:
	- O núcleo de um requisito é a funcionalidade que é especificada. Exemplo: Salvar, imprimir, calcular e etc.
	- Esta funcionalidade é chamada de processo.
	- Processos são atividade e só pode ser descritas com verbos.
	- Neste caso descrevemos o processo que descreve o comportamento do sistema.
	- As palavras que utilizam para descrever o processo definem a semântica e, portanto, precisam ser claros e consistentes.

Passo 3. Caracterizar a atividade do sistema:
Para requisitos funcionais a atividade do sistema pode ser classificada em três tipos:
	- Atividade autômoma do sistema: o sistema executa um processo de forma autônoma.
	- Interação do usuário: o sistema fornece o processo ao usuário como um serviço.
	- Requisito de interface: o sistema executa um processo dependendo de terceiros; (EX: outro sistema). O sistema é passivo e aguarda por um evento externo.	

Template tipo 1: Atividade Autônoma de Sistema
	- Utiliza-se o tipo 1 quando requisito são construídos para descrever atividades do sistema que são executadas de forma autônoma.
	- O usuário não interage com a atividade.
	- O verbo neste caso descreve a funcionalidade executada pelo sistema.

Template tipo 2: Interação com usuário
	- Este é o tipo de template utilizado quando o sistema fornece uma funcionalidade para o usuário onde o usuário faz input de dados ou quando o sistema interage diretamente com o usuário.

Template tipo 3: Requisito de Interface
	- Este é o template utilizado quano o sistema executa uma atividade e é dependente de terceiros.
	- Sempre que mensagens ou dados são recebidos por um terceiro ( ex: outro sistema) o sistema tem que reagir e executar um comportamento específico.

Passo 4. Inserir objetos e complementos:
	- Alguns verbos exigem um ou mais objetos para serem considerados completos.
	- Esses complementos podem ser adjetivos e advérbios.
	- O verbo imprimir, por exemplo, necessita de o quê está sendo impresso e onde está sendo impresso.

Passo 5. Determinar condições lógicas e temporais
	- Requisitos geralmente não documentam funcionalidades contínuas.
	- Requisitos geralmente documentam funcionalidades  ou fornecidas sob certas restrições lógicas e temporais.
	- Deve-se evitar a utilização da palavra "Quando."
	- Para diferenciar condições lógicas de condições temporais utilizamos as seguintes construções:
		- Assim que, para condições temporais
		- Se, para condições lógicas

 -  Template de requisitos devem ser utilizados quando membros do projeto mostram interesse em um processo de desenvolvimento formal.
 -  Estilo e criativadade são limitados quando templates de requisitos são utilizados. 
 -  A utilização de templates de requisitos não deve ser feita de forma compulsória, mas deve ser adotada como ferramenta suplementar.

Resumo: 

Linguagem natural é frequentemente usada na documentação de requisitos, (prosa livre).
Como vantagem: Não exige tempo de preparo para ser lida e compreendida pelos stakeholders.
Não existe limites para documentar, documenta-se qaisquer circunstâncias usando a Linguagem Natural. 
Desvantagem: A linguagem natural é inerentemente ambígua, podemos muitas vezes ser interpretadas de várias formas pelas partes interessadas, por seus diferentes níveis de conhecimento, origens sociais e diferentes experiências.

Ocorre os chamados efeitos transformacionais:

	- Nominalização
	  - Definir os processos completamente
		- Ex: (input, reset, crash)
	- Substantivo sem indicador de referência
	  - São frequentementes especificados de forma incompleta ou incorreta
		- Ex: (o usuário, o sistema, os dados ou a função)
	- Quantificadores universais
	  - Especificam quantidade e frequência.
		- Ex: (nunca, sempre, nenhum, cada, todos, alguns ou nada)
	  - Agrupam um conjunto de objetos e fazem declaração sobre  comportamento deste grupo.
	- Condições especificados de forma incompleta
	  - Condições especificam o comportamento que deve ocorrer quando a condição não for atendida
		- Ex: (Se... Então, caso... se, ou; depedendo de...)
	- Verbos de processo especificados de forma incompleta
	  - Alguns verbos requerem mais de um substantitivo, como o transmitir, o que é transmitido, para onde é transmitido.
	  - Evitar usar a voz passiva.
		- Uso e voz ativa.

Na construção de templates de requisitos tem que seguir os passos:
vantagens: apoio a alta qualidade e não-ambiguidade sintática, baixo custo e tempo otimizado.

Passo 1 - Determinar a obrigatoriedade legal:  Usando os verbos modais (DEVERÁ, DEVERIA, IRÁ)

Passo 2- Determinar o núcleo do Requisito: Funcionalidade ( Processo).

Passo 3 Caracterizar a Atividade do Sistema: Atividade autônoma; Interação com o usuário; Requisito de Interface
	Template tipo 1
		<Verbo de processo> usado para forma autônoma do sistema
	Template tipo 2 - Interação com o usuário
		Sistema fornece ma funcionalidade paraum usuário, ou interage diretamente com o usuário.
	Template tipo 3 - Requisito de Interface
		Sistema executa uma atividade e depende de sistemas adacentes.

Passo 4 - Inserir objetos e complementos: Alguns verbos de processos necessitam de mais de um ou mais objetos adicionais para ser considerados completos.

Passo 5 - Determinar Condições Lógicas ou temporais: Requisitos não docuementam funcionalidade contínuas, mas funcionalidades que são executadas ou fornecidade somente sob certas condições, lógicas ou temporais. Usando conjunção lógica "SE" , conjunção temporal "ASSIM QUE", evitar o uso de quando.

Templates devem se usados quando membros do projeto tem interesse em um processo formal de desenvolvimento. Estilo e criatividade são limitados quando templates de requisitos são utilizados.Usar como ferramenta complementar.
